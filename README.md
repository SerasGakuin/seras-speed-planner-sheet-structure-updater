「シート構成一括更新」マクロ　仕様書

すべての生徒の「スピードプランナー」のシートの構成を一括で更新し、データの移行もするためのマクロです。
注意点！

特定範囲のフォントサイズを変更したい、などの軽微な変更については、月末月初マクロの「UpdateOnGesho」機能をご利用ください。

本マクロはシートを丸ごと新しいものに交換する、大掛かりなものです。
テンプレートからシートごとコピーすることにより、更新を実行します。
いざとなれば手動で現実的な手間の大きさで修正できるようなバックアップを取ることもサポートします。

コード上で一つ一つ変更内容を定義していると項目が複雑でミスが誘発されるので一気に交換したい
GASのコードだけでは仕様上設定できない部分を更新したい

といったケースでのみご利用ください。


また、本マクロには冪等性がありません！
旧シートから新シートへのデータの移行ロジックが、それぞれのシート構成に依存しており、バージョン管理のような仕組みがまだないためです。

通常使用時に操作していただくのは、Configs.gsと、API.gsのみです。

Configs . gs
こちらは、更新実行時のもろもろの設定項目を定義する部分です。
設定項目は、コード内にコメントでの説明があります。
おおよそ、更新対象のシートと、新シートへのデータの移行方法と、更新対象から特別に外すブックの設定です。

API . gs
こちらは、手動で実行する関数が定義されているファイルです。通常の使用時に書き換えは不要です。

本番更新実行

テスト用ブックでの意図通りの更新を確認したら、本番の実行に移ります。
テスト用の時に更新したシートの名前と、sheetNamesの配列の中身が一致することを確認してください。これらのシート名が本番用キューの構築時に使用されます。


もし、事前に手動更新したなどの事情で二重に更新することを避けたい、などで更新対象にくわえることを避けたいブックがある場合は、willNorUpdateBooksUrlの配列にurlを追加してください。このurlは、url全体からid部分を抽出することによって利用するので、/edit以降になにか付いていても問題ありません。
（万が一間違えて更新実行しても、バックアップシートは作成されます。このシートは更新実行前の状態をそのまま保持しています。）


（このとき、設定に明らかな不整合があれば、キューをセットする前にある程度検出し、エラーをだす自動チェックを追加しましたが、完璧ではありません。）



そのほかの設定は更新に成功したテスト時の設定をそのまま利用してください。

ここまで確認したら、buildUpdateQueues()を実行して下さい。キューシートに自動で更新対象のブックとシート名の直積のキューが構築されます。



次にprocessUpdateQueues()を実行してください。これによって本番の更新が開始されます。トリガーによるバッチ処理を実装しているので、通常の実行時間制限(6分)を超えて、長時間実行されることが多いです。一時間以上かかることもありますので、業務の終わりに実行するのが良いかもしれません。

更新の仕組み

「シート構成参照用スピードプランナー」のシートを生徒のスピードプランナーにコピー。
もともとそこに存在していたシートをコピーしてバックアップを取る。
もともとそこに存在していたシートそれ自体（2で作ったバックアップでない方）を削除
2で作成したバックアップから、Configsの設定に従ってデータを、1でコピーしてきたシートにコピー（移行）
3でオリジナルを削除したことで#REF!エラーを発生させている数式をリフレッシュ（再読み込み）
これを各対象シートに対して実行

3でオリジナルをわざわざコピーしてから原本を削除するのは、単純なリネームでバックアップを取るとオリジナルを参照していた数式内の参照名も自動で追従して変更されてしまい、修復困難になるからです。
リネームでなく削除であれば、数式内の参照名には変化がありません。ただし、内部キャッシュの問題で参照エラーになるので、新しいシートをもってきてオリジナルの名前に変更してから、数式をリフレッシュするのです。
そうすれば数式は新しいシートの方を参照します。

その他補足(設計思想・エラー対応)

QueueConsumerが本体で、Builderのほうは補助です。手間を度外視すれば、キューの構築は手動でも構いません。



生徒マスターから更新対象を検索するので、そちらで「在塾」状態でスプレッドシートのurlが登録されていないと更新されません。ログに失敗が記録されておらず、かつ更新されていないようならば、更新対象から外れていて、古い状態で残されているだけである可能性が高いので、数件ならば手動で更新するのが確実で早いと思います。このマクロは「冪等性」がないので




更新途中にエラーが発生した場合にも、データはどこかのシートには残るようになっています。
シートの状態の可能性は以下:

1. 更新成功している（新しい構成のシートと、バックアップの両方にデータあり）（いまのところこれ以外経験していない）

2. バックアップと古いシートだけあって新しいシートなし(両方にデータあり)（更新していないのとほぼ同じ。バックアップだけ最新のものになっているが、手動でシートの更新が必要。）

3. バックアップだけあり（バックアップにデータあり。手動でシートの更新が必要。）

4. バックアップと空の新しいシートだけあり（バックアップにデータあり。手動でデータ移行して、エラーの原因が通信障害などの不可避のものでないなら修正してもう一度API.gsから実行開始すればよい）

5. 新しいシートにデータが移行されており、バックアップもあるが、数式が複数個所で#REF!になっている（移行は完了しているので、フォーマット修正マクロで数式をリフレッシュすれば完了します。）
